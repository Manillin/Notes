# RKE2 - Kubernetes Distribution   

RKE2 è una distribuzione Kubernetes di SUSE/Rancher che ha come particolarità la semplicità di installazione e le politiche di sicurezza.  



/etc/rancher/rke2/rke2.yaml: È il Kubeconfig. Viene generato automaticamente da RKE2 dopo l'accensione. Contiene i certificati crittografici di cluster-admin. È letteralmente la chiave del cluster: chi lo possiede può controllare tutto. Per comandare il cluster dal pc locale scarichiamo questo file localmente e lo esportiamo come: export KUBECONFIG=path/to/rke2.yaml   


### Operazioni di configurazione con Jinja2  

/etc/rancher/rke2/config.yaml: È il Blueprint (il progetto). Si crea prima di avviare il servizio.   Dice a RKE2 come deve configurarsi quando si accende.  
Lo creeremo a partire dal file di configurazione template dinamico di jinja2   

Bisogna prestare attenzione ai permessi del Kubeconfig (rke2.yaml)   
Di default, RKE2 crea il Kubeconfig con permessi `0600` (visibile solo a Root) per sicurezza. Impostando `write-kubeconfig-mode: "0644"` nel `config.yaml` (quello che generiamo a partire dal template di jinja2), diciamo a RKE2 di assegnare i permessi di Lettura a tutti gli utenti del sistema. Questo ci permette di usare Ansible (o un normale utente SSH) per scaricare il file sul nostro PC locale senza incappare in errori di permessi.   


tls-san (Subject Alternative Name): L'API di Kubernetes genera un certificato TLS/SSL che, di default, certifica solo l'IP privato del Master. Se proviamo a collegarci al cluster passando per l'IP di un Bastion, la connessione verrà rifiutata per "Certificato non valido". Aggiungendo l'IP pubblico del Bastion alla lista tls-san nel `config.yaml`, istruiamo RKE2 a includere quell'IP nel certificato, dichiarandolo "fidato".  
Faremo questa operazione sempre dal template jinja2  


## Installazione Singolo Nodo Master (Non-HA)  

Si descriverà l'installazione di un control plane RKE2 su una singola macchina virtuale.  

### 1. Preparazione delle directory:  
RKE2 richiede una struttura di directory specifica per leggere la sua configurazione prima dell'avvio, la cartella non esiste di default e va creata!   
- creiamo con ansible `/etc/rancher/rke2`  


### 2. Creazione del Blueprint (config.yaml)  
Prima di installare il binario, dobbiamo definire lo stato desiderato del cluster.  
Definiamo lo stato tramite il file `/etc/rancher/rke2/config.yaml`   

Solitamente generiamo questo file tramite template jinja2, qui dobbiamo anche impostare e iniettare l'IP corretto del loadbalancer/bastion.  
Impostiamo anche i permessi più larghi per permettere la copia del file

```
# Esempio di /etc/rancher/rke2/config.yaml
write-kubeconfig-mode: "0644"
tls-san:
  - "<IP_PUBBLICO_BASTION>"
```

### 3. Installazione del binario:  

RKE2 fornisce uno script di installazione ufficiale che scarica i binari corretti e predisponde i servizi `systemd`, senza però avviarli automaticamente.  

```bash
curl -sfL https://get.rke2.io | sudo sh - 

```
(Ovviamente tale opreazione verrà fatta da Ansible)   

### 4. Avvio del servizio:   

Una volta installato rke2, abilitiamo il sevizio per l'avvio al boot della macchina e lo accendiamo.  
*Nota*: il primo avvio può richiedere diversi minuti (2~5) poichè rke2 deve scaricare le immagini dei container core (coreDNS, etcd, Ingress) e generare tutti i certificati scrittografici.   

Verrà fatto dal playbook ansible:

```ansible
- name: Abilita e avvia il servizio rke2-server
  service:
    name: rke2-server
    enabled: yes
    state: started
```

### 5. Verifica Interna:  

Per verificare che rke2 sia stato installato che stia girando e che il nodo master sia in stato `Ready` interroghiamo l'API-Server direttamente dal nodo stesso.  
Utilizzeremo `kubectl` e il `kubeconfig` generati da rke2.  

Entriamo in ssh dal bastion e proviamo:

```bash
sudo /var/lib/rancher/rke2/bin/kubectl --kubeconfig /etc/rancher/rke2/rke2.yaml get nodes
```

### 6. Configurare l'accesso remoto  

La best practice impone di amministrare il cluster dalla propria postazione locale, non collegandoci in ssh al nodo.    

Per farlo dovremo: 
- copiare il `kubeconfig` dal master al proprio PC (fatto tramite Ansible)  
- modifica dell'IP: il file originale indica l'API-Server su `127.0.0.1` ; poichè ci troviamo all'esterno della rete privata, dovremo aprire tale file e sostituire l'indirizzo `127.0.0.1` con il FIP del bastion (che agisce da LB TCP lv4 inoltrando il traffico sulla porta 6443)  
  - Questo lo facciamo da ansible con una  task `fetch` e con una `regexp`  

- esportiamo la variabile `KUBECONFIG=~/.kube/config` passandogli il path dove abbiamo messo il `kubeconfig` che abbiamo precedentemente copiato in locale.  


### 7. Controllo locale  

A questo punto potremo comandare il cluster dal pc locale, per fare una prova basterà fare:  

```bash
kubectl get nodes
```  
e ci aspetteremo un output di questo tipo:  
```bash
cvonwaldorff@NCVONWALD204771:~$ kubectl get nodes
NAME            STATUS   ROLES                AGE   VERSION
test-worker-1   Ready    control-plane,etcd   20h   v1.34.6+rke2r3
```
