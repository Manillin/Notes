# Terza Lezione K8s


Ambiente di sviluppo -> l'unico modo per capire le tecnologie è USARLE   

L'ambiente di sviluppo utilizzato sarà Bare Metal, quindi si vedrà un caso generale e complesso.   

Host: Fedora 42+ ; QEMU + KVM  

Layout del cluster:
- 3 nodi server 
- 2 nodi worker 
- è l'architettura minima di produzione consigliata  

Container Runtime Interface -> CRI-O (container runtime mminimale progettata espressamente per k8s e nient'altro)  


Container Network Interface -> kube-router (semplice e privo di addon rispetto a k8s vanilla)  


Container Storage Interface -> NFS CSI Plugin; lo storage non ad alta affidabilità che installiamo sul primo nodo (installeremo uno share di rete sul primo nodo, tutti gli altri nodi monteranno questo share di rete; è un file storage distribuito)    


Metodo di installazione:  

kubeadm -> unico metodo ufficile supportato upstream   


Control Plane Endpoint:  

L'endpoint del control plane di k8s deve essere esposto in modo highly available (deve prescindere dai nodi server).    

Metodo raccomandato dalla documentazione per High Availability: keepalived + haproxy (fornisce load balancing e failover)   

Il service discovery avverrà via DNS SRV record: control plane flessibile (in questo modo se dobbiamo aggiunere o togliere un nodo basta cambiare i record DNS, non serve cambiare la config del load balancer)     


L'endpoint del control plane è definito per nome: possibile cambiare gli IP senza invalidare i certificati

**Preparare i nodi**:  
I controlli inziali sono:  
1. MAC address univoco per ogni nodo (iplink)
2. Hardware ID univoco per ogni nodo (`sudo cat /sys/class/dmi/id/product_uuid`)
3. Swap disattivato(se il nodo ha 32Gb di RAM il nodo deve vedere solo quei 32Gb)
4. Hostname univoco (`cat /etc/hostname`)
5. Alias a localhost in `/etc/hosts`
6. Adiacenda L3 con gli altri nodi (tutti i nodi possono essere contattati a un indirizzo ip)   



**Installare la CRI (in questo caso CRI-O):**   
1. installare gpg
2. download delle chiavi d firma 
3. repository di cri-o
4. apt update 
5. apt install cri-o
6. systemctl enable --now crio.service 

**Control plane endpoint**   
Dobbiamo dare una identità al nostro control plane  
anche se possiamo usare gli IP, si consigli sempre un nome DNS, e in ambiente metal mettiamo:  
- un load balancer su ogno nodo server 
- un indirizzo IP flottante condiviso tra i nodi server 
- un nome DNS associato all'IP flottante 

`keepalived` permette di condividere indirizzi ip tra più host, usando un protocollo chiamato VRRP (virtual router redundancy protocol)    

`haproxy` è il loadbalancer che ridistribuirà il carico agli API server (apt install hproxy)    
ha un fronted (dove ascolta il load balancer) e un backend (dice quali servizi devo colpire, implementa un DNS-RR)

## Installazione Orchestrata   


Ansible : strumento per automatizzare operazioni massive sui nodi  
è un sistema agent-less: lancia comandi ssh senza installare nulla    


Ansible segue il seguente flusso:
- viene deinita la lista degli host ossia la lista di dove vogliamo lanciare i nostri comandi (file inventory)
- viene definito un **playbook** (lista di ruoli da applicare a host nell'inventario) (lista di cose che vengono fatte una dopo l'altra)
- viene eseguito il playbook   


Elementi di un **ruolo**: (il ruolo è l'elemento del playbook)   
- una serie di task, lanciati in ordine
  - I task sono "pezzi" di Ansible (funzioni di libreria che si chiamano *moduli*) che possiamo lanciare con argomenti.  
  - I task possono avere modificatori (es. essere resi condizionali)
- una serie di file template   

```yaml
- hosts: k8s-servers, k8s-workers
  become: true 
  vars:
    k8s_version: 'v1.34'
    cri_o_version: 'v1.34'
  any_errors_fatal: true 
  roles:
    - role: install-cri-o
    - role: instsall-k8s
```

I ruoli vengono fatti tutti in parallelo, alcune operazioni però devono essere sequenziali (es. collegare un nuovo nodo -> serve token lato server e poi fare auth)   

Ansible permette di fare questa coas usando il parametro `serial` (da inserire ad esempio nella fase di bootstrap di k8s)
