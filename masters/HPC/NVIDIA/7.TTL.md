Luster - high performance 
NFS (NAS?) - base (low performance) 


in un cluster production level la home directory è su un file system low performance come NAS (non si invitano gli utenti a usare la home dir come HP storage).  

Module files -> settano variabili di ambiente per farle puntare a uno specifico sw environment   
LD_LIBRARY: librerie
CPATH e C_INCLUDE_PATH: header files   
Si possono creare autonomamente, nvidia fornisce i default tramite nvhpc-hpcx 
- hpcx ha solo le communication libraries 
- nvhpc ha communication libraries + tutto il sw stack per far girare l'applicazione 
per caricare un modulo: `ml load nvhpc/25.5`    
per vedere la lista di module file caricati in un determinato momento: `ml list`
per pulire l'ambiente: `ml unload nvhpc/25.5`

I module file sono un grande vantaggio rispetto all'uso di export PATH=/....

nvidia-smi topo -m: comando utile in caso di sistema multi gpu, fa il display di come sono collegate le GPU con le interfacce di rete, serve per controllare i percorsi, vogliamo evitare che una applicazione prenda un percorso non ottimale (idealmente dovremmo usare gli nvlinks)

deployment di un cluster non avviene manualmente, ma tramite dei progrmami, in maniera automatica.  
si spara una versione base dell'os sui nodi (con minimo di config di rete) e poi si scrivono ansible playbooks, puppet o altro.   
si fa il deployment di tt le config con ansible (o alternative) during boot time, quando il sys fa il boot dell'OS (prima di arrivare alla shell del login), queto è il momento in cui vengono applicati gli ansible playbooks -> configurare il tpserver,  dhcp, install lib base, modulefiles.   


Questo perchè sempre piu spesso si trovano cluster che sono 'Stateless', l'OS non viene istallato in un hard disk ma in memoria , vuol dire che tutte le volte che facciamo reboot il nodo si reinstalla tramite ansible -> piu veloce.  

quando il sistema è up and running e dobbiamo installare una libreria su tutti i nodi perche non vogliamo fare il reboot di tutti i nodi allo stesso tempo (o controllare se tutte le porte sono alla speed che ci aspettiamo), ci sono programmi come 'pdsh'.  

Consente di far girare in parallelo un comando su tutti i nodi del cluster (o su un subset di nodi), evitando di dover fare ssh.   


pdsh -R ssh -w lego,...  date (controlla la data su tutti i nodi specificati).  

pdcp invece funziona per copiare file tra nodi (tipo file di config, dal nodo 1 al nodo 2 true20)

OSU - bandwidth e latency multinodo:  


./configure --prefix=/home/nvidia/ftartaglione/osu-7.4-patched_25.9 CC=/home/nvidia/ftartaglione/hpc_sdk_259/Linux_aarch64/25.9/comm_libs/hpcx/bin/mpicc CXX=/home/nvidia/ftartaglione/hpc_sdk_259/Linux_aarch64/25.9/comm_libs/hpcx/bin/mpicxx --enable-cuda --with-cuda-include=/home/nvidia/ftartaglione/hpc_sdk_259/Linux_aarch64/25.9/cuda/12.9/include --with-cuda-libpath=/home/nvidia/ftartaglione/hpc_sdk_259/Linux_aarch64/25.9/cuda/12.9/lib64    

-> comando per la build di OSU

 

--prefix per decidere dove vogliamo che vengano installate le cose (optional ma è consigliabile! ed è sempre suggerito installare software dove non hai sorgenti) 

CC= definisce il compilatore C/C++ , semplicemente un ptr alla versione di MPI (in quanto questo è un mpi benchmark).  
Per avere mpi basta installare nvidia hpc sdk che ha mpi dentro le com libs.  

CXX= la stessa cosa di CC ma con mpicxx   

Poi si attiva CUDA e si mettono i ptr agli header file e ai library path


seguito da make e da make install che fa quello che ci serve.  


dopo avremo il folder con dentro gli eseguibili, divisi per categorie, le piu importanti sono p2p e dentro a quelle ci sono le OSU bandwidth e OSU latency -> (osu_bw, osu_latency).     

Risultati di OSU da analizzare -> puntare ad avere High bandwidth e low latency (per network, in particolare per infiniband)  
Ci concentriamo su una size di messaggi alta per controllare la bandwidth (nell'esempio avevamo 24763 MB/s con OSU e facendo ibstat potevamo vedere di avere 200 Gb di banda.)


200 Gb   -> banda disponibile, risultato con ibstat
24763 MB -> 25GB -> 200 Gb risultato di OSU

-> conclusione: la banda è a posto !    

Quando guardiamo la latency con OSU dobbiamo puntare ad avere valori bassi! il più basso possibili.  
se la rete è sotto 2 (microsecondi) vuol dire che è OK!
