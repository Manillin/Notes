## OpenStack basics   

- Identità (Keystone)
- Networking (Neutron)
- Compute - VMs (Nova)
- Storage (Cinder/Manila)


Ogni nodo fisico del cluster ha un OS linux dove sopra gira **KVM** che fa da Hypervisor (quasi) di tipo 1.    
Openstack viene installato sui nodi e vive nel loro OS, è un insieme di servizi che invia ordini ai nodi parlando con KVM.   

Si astrae il compute del cluster -> tutti i nodi del cluster diventano un unico pool di risorse   

Dentro i nodi fisici nasceranno le VM con il loro OS e dentro a queste VM si installerà K8s.   
I vincoli hardware per la creazione di VM sono:  
- RAM: è una risorsa incomprimibile, se il nodo fisico del cluster ha 50GB di RAM, non possiamo assegnare più di tale quota alle VM creeremo in tale nodo. 
- CPU: vincolo che consente maggiore flessibilità, è abilitato l'overcomitting. è consentito assegnare più vCPUs di quelle che si hanno fisicamente sul nodo. Finchè le VM non usano tutte il 100% della CPU contemporaneamente, il sistema regge  
- Distribuzione: Lo Scheduler (componente openstack) decide dove piazzare le VM, cercherà di distribuirle sui nodi fisici più liberi per garantire alte prestazioni.

### Step2 -> creiamo la rete underlay (neutron)  
si crea la network privata, una subnet e un router virtuale che collega la rete privata con quella pubblica per fare uscire le macchine.  


resource "openstack_networking_network_v2" "network_1" {
  name           = "network_1"
  admin_state_up = "true"
}

crea uno switch 'vuoto' che rapprensenta la rete e lo accende (crea un dominio di broadcast isolato, livello2).  
Definiamo qui dentro il nome della nostra rete privata.  

resource "openstack_networking_subnet_v2" "subnet_1" {
  network_id = openstack_networking_network_v2.network_1.id
  cidr       = "192.168.199.0/24"
}

Prende lo switch vuoto che abbiamo creato prima per iniettarl la logica di livello3, passandogli l'id della rete / switch che abbiamo creato prima.   
in cidr definiamo il range per i nostri indirizzi IP (genera il pool di IP)   
Da questo momento la rete esiste ed è pronta.  

resource "openstack_networking_router_v2" "router_1" {
  name                = "my_router"
  admin_state_up      = true
  external_network_id = "f67f0d72-0ddf-11e4-9d95-e1f29f417e2f"
}

Gli si da un nome e soprattutto l'id della rete esterna. Il router prende i pacchetti della rete interna, ci mette l'IP pubblico della rete esterna e li manda su internet (fa il percorso inverso quando arriva la risposta)  

resource "openstack_networking_router_interface_v2" "router_interface_1" {
  router_id = openstack_networking_router_v2.router_1.id
  subnet_id = openstack_networking_subnet_v2.subnet_1.id
}

Serve per collegare la subnet al router, senza sarebbero due componenti funzionanti ma senza la capacità di interagire.   

resource "openstack_networking_floatingip_v2" "floatip_1" {
  pool = "public"
}

Diciamo a OpenStack da dove pescare l'IP flottante.  
Il floating IP è un indirizzo IP pubblico, statico e riservato che OpenStack aggancia a un IP privato specifico.  



### Step3 -> Sicurezza e Accessi (security grops e KeyPairs)  
si creano le chiavi ssh per entrare nelle VM (keyPairs)  
si scrivono le regole del firewall(security groups). aprire la pora 22(ssh), la 6443(api k8s), le porte della CNI (VXLAN 8472) e le porte web (80/443).   

resource "openstack_compute_keypair_v2" "test-keypair" {
  name       = "my-keypair"
  public_key = file("~/.ssh/id_ed25519.pub")
}

Carichiamo la nostra chiave pubblica SSH, openStack la salva e la userà in futuro quando vorremmo collegarci alle VM. se non la definiamo non potremo mai collegarci alle VM e saremo chiusi fuori.  
Diamo un nome leggibile alla chiave, dando un valore alla variabile 'name'    


resource "openstack_networking_secgroup_v2" "secgroup_1" {
  name        = "secgroup_1"
  description = "My neutron security group"
}

Crea un contenitore vuoto che corrisponde a un profilo di sicurezza vuoto: security group.     
Serve perchè in OpenStack non si applicano le regole del firewall alle singole macchine! si crea un security group e lo si applica alle macchine.  
se un domani vogliamo aprire una nuova porta, non dovremo andare su tutti i nostri nodi e modificare il firewall, modificheremo solo il security group e la regola verrà applicata a tutti i nodi che ci appartengono.    


resource "openstack_networking_secgroup_rule_v2" "secgroup_rule_1" {
  direction         = "ingress"
  ethertype         = "IPv4"
  protocol          = "tcp"
  port_range_min    = 22
  port_range_max    = 22
  remote_ip_prefix  = "0.0.0.0/0"
  security_group_id = openstack_networking_secgroup_v2.secgroup_1.id
}

Corrisponde alla regola vera e propria del firewall.  
Ricordiamo che di default la policy è Deny All! quindi dovremo scrivere le regole per aprire il traffico interno (es. porta22 per ssh, porta 6443 per parlare con api-server, ...).   
la direction mi specifica la direzione -> ingress o egress   
port_range_min e port_range_max mi definiscono il range di porte che mi interessa abilitare. 
passiamo anche il security group! questa regola verrà applicata a tutti le VM che appartengono a tale gruppo.   
il remote_ip_prefix definisce chi può entrare (0.0.0.0/0 significa tutti).  
Es. per permettere la comunicazione con api-server k8s mettiamo in remote_ip_prefix la rete privata con la cidr che abbiamo definito prima.   




### Step4 -> le macchine virtuali (Nova)   
si definiscono le risorse di calcolo, scegliere il flavor (taglia della VM) e l'image (OS).  

resource "openstack_compute_instance_v2" "basic" {
  name            = "basic"
  image_id        = "ad091b52-742f-469e-8f3c-fd81cadf0743"
  flavor_id       = "3"
  key_pair        = "my_key_pair_name"
  security_groups = ["default"]

  metadata = {
    this = "that"
  }

  network {
    name = "my_network"
  }
}

Il campo metadata serve per aggiungere dei tag alle macchine nel database di openstack, può tornare utile per fare query mirate.  


**cloud-init**  

resource "openstack_compute_instance_v2" "instance_1" {
  name            = "basic"
  image_id        = "ad091b52-742f-469e-8f3c-fd81cadf0743"
  flavor_name       = "nome_del_flavor"
  key_pair        = "my_key_pair_name"
  security_groups = ["default"]
  user_data       = <<-EOF
    #!/bin/bash
    echo "Inizio la configurazione della macchina..."
    apt-get update -y
    apt-get install -y curl vim
    curl -sfL https://get.rke2.io | sh -
    echo "RKE2 scaricato con successo!"
  EOF

  network {
    name = "my_network"
  }
}

Possiamo creare una macchina virtuale e usare il campo `user_data` per passare uno script!  
Nelle immagini Linux del mondo cloud è installato un programma chiamato cloud-init, appena si avvia una macchina di questo tipo per la primissima volta (e solo per la prima volta), cloud-init si sveglia, prende da openstack lo script definito in `user_data` e lo esegue con privilegi di root.  

*Nota* user-data non sostituisce ansible, ma può facilitare e preparare le macchine per lavorare meglio e facilitare l'installazione di k8s (che avverrà sempre tramite ansible!!)
