### Registro privato    



Serie di nodi che possono schedulare pod che contengono container che vengono eseguito a bordo di tutti i nodi del nostro sistema.  
le immagini quindi sono il cuore del sistema -> tutti i nostri container sono creati a partire da immagini.    

Abbiamo bisogno di un luogo dove immagazzinarle -> il registro privato   

---  

Registro: è un servizio offerto tramite http che funge come magazzino di immagini OCI identificati da un nome e un tag.  

Un esempio pubblico è dockerhub  

docker.io/posgres:18 -> hostname / name_segment (nome dell'immagine) : tag (puntatore a un immagine OCi in un certo momento della storia)    


l'immagine sottostante può cambiare, il nome e il tag sono dei puntatori, una sorta di alias.    


Scaricare un immagine da un registro consiste in:  
- richiesta per il manifest associato al nome e tag 
- richista di tutti i layer presenti del manifest  

Tutte queste richieste sono autenticate dal registro e il registro è sostanzialmente stateless.    


--- 

Un registro PRIVATO in un cluster serve perchè: 
- cache in locale delle immagini per:
  - mitigare li effetti dei limiti imposti dai registri upstream (es. quante immagini possiamo pullare in 1h)
  - aumentare l'availability (se amazon aws va down non possiamo pullare immagini)
- archivio delle nostre immagini 

tenere un registro privato ci permette di andare più veloci e di avere una sorta di stato persistente per le nostre immagini.  

--- 

Autenticazione:   

Per accedere a un registro privato, il kubelet o la CRI runtime devono esser informati delle credenziali da utlizzare.  
Se il nostro registro usa certificati TLS self signed o è esposto via HTTP, va configurato anche l'accesso insicuro.   


### Configurare le credenziali tramite CRI:   

Il primo modo per fornire delle credenziali è: in fase di configurazione del cluster, di configurare la CRI con delle credenziali precaricate  

vantaggio -> semplice da fare    
svantaggio -> procedura che dipende dalla CRI, non permette di proteggere i contenuti del registro (tutti gli utilizzatori del cluster possono accedere a tutte le immagini)   

è consigliato per le immagini "fondamentali" del sistema   


### ImagePullSecret  

consiste nel creare un ImagePullSecret, un secret che pò essere poi utilizzato da un pod.  

come funziona -> creiamo un pod, gli diciamo che imagePullSecret sta usando questo pod, quando il pod viene schedulato il kubelet se ne accorge e scarica l'imagePullSecret e dice alla CRI di usare le credenziali che qualcuno ha dato prima di usare le credenziali built in.   

Vantaggi: non dobbiamo riconfigurare il cluster e possiamo utilizzare credenziali arbitrarie e di segmentare l'accesso.  

svantaggi: dobbiamo ricordarci di includere il pull secret in tutti i nostri pod. i secret sono namespaced quindi potremmo doverne creare molti (uno per namespace)  

è consigliato per le immagini delle applicazioni.    


```bash
kubectl create secret docker-regiistry demo-ips \
    --docker-server=registry.k8scourse.serics.eu \
    --docker-username=registry-user \
    --docker-password=registry-password \
    --docker-email=registry@k8scourse.serics.eu \
    --dry-run=client -o yaml > secret.yaml 
```

### Kubelet ImageCredentialProvider  

Questo metodo consiste nel configurare su ogni nodo un plugin che ricevendo una richiesta la analizza e fornisce in output se esistono credenziali disponibili.  

Vantaggi: permette di usare credenziali arbitrarie e volendo segmentando anche per service account (service account -> carta di identità del pod)     

Nota: [Service Account] Ogni Namespace ha un Service Account chiamato default. Se tu non specifichi nulla nel file YAML del Deployment, il Pod nasce con quella carta d'identità "base" che solitamente non ha permessi.  
Si possono creare ServiceAccount personalizzati e assegnare permessi tramite RBAC.  

Svantaggi: dobbiamo implementarlo, va configurato il kubelet e installato il plugin su tutti i nodi  

Consigliato per setup avanzati   



### Considerazioni in produzione  

Il registro immagini è un servizio critico per un cluster, se un immagine è assente e il kubelet non la ha nella cache, il container non può essere avviato.     

Le immagini di solito sono molto grandi (~GB) e il registro è *Storage Instensive*.  
Dobbiamo quindi anche preoccuparci di rimuovere periodicamente le immagini che il nostro cluster non sta più usando dal registro per liberare spazio.  

Il min time to recover di un Pod è influenzato dalla velocità del registro!    

*Per vedere tutte le immagini del cluser*:    

```bash
kubectl get pods --all-namespaces \
-o jsonpath="{.items[*].spec['initContainers','containers'][*].image}" | \
tr -s '[[:space:]]' '\n' | \
sort | \
uniq -c 
```

### Accesso al Cluster  

Autenticazione: ricevere una decisione che permette di associare una data richiesta ad una identità (o più di una)   

Identità in termine di Kubernetes:  
Per l'API-Server di K8sl'identità ricade in una di queste famiglie:
- richiesta anonima (non autenticata)
- service accounts (un identità di un entità del cluster)
- user request (un identità di un entità esterna al cluster)

Flussi di autenticazione:  
In k8s tutto passa dall'API server, quindi l'autenticazine consiste in operazioni da efettuare sulla richiesta stessa, come aggiungere header e simili  

In particolare K8s usa come sue primitive di autenticazione:
- l'aggiunta di un bearer token nell'header HTTP "Authorization"
- La presentazione di un certificato X509  

Autorizzazione:  

Dopo essere stati autenticati, ci si domanda cosa posso fare -> questa è la procedura di autorizzazione.    

è l'operazione più sensitive -> quello che possiamo fare dentro una volta autenticati al cluster corrisponde alla superficie di attacco del nostro sistema.      


Il modello predefinito di K8s è **Role Based Access Control (RBAC)**     

I permessi sono definiti in dei ruoli e le identità sono collegate ai ruoli tramite bindings.    


Il modello opzionale (e sconsigliato) è *Attribute Based Access Control (ABAC)*   
molto granulare ma difficile da utilizzzare e mantenere.   

**Politiche**:   
In K8s la policy di default è un generico **deny all**.  
Questo vuol dire hce ogni operazione va autorizzata separatamente!  


Fornisce protezione di default 


### Autenticazione e autorizzazione in K8s.   

**Role e ClusterRole**  

L'oggetto APi che ci permette di definire dei ruoli e delle politche si chiama Role (versione namespaced) o cluster role (versione non namespaced - aka si applica a tutto il cluster)   

Role e Cluster role sono identificati da un nome e contengono all'interno una lista di regole.     

sono delle risorse, come se fossero dei pod (posso fare get, delete, ...)    


**Regole di autorizzazione**   
Una regola specifica quali azioni 'verb' possono essere fatti su che risorsa 

```yaml
- apiGroups: [""] # default v1
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
  reourceNames: ["my-pod"]
```

- apiGroups: dice APi version, nel caso v1 (default) è la stringa vuota
- possiamo usare kubectl api-resources per vedere l'api group e il nome delle resources 
- possiamo anche definire policy su risorse definite per nome elencodolo sotto "resorceNames" - nell'esempio potrebbe fare quei verbs solo su quello specifico pod con quel nome  


dove prendo i verbs? come faccio a sapere quali verbs sono definiti sulle mie risorse?
- cercare nell'api reference (documentazione) la risorsa che ci interessa e andare a vedere le operazioni che possiamo svolgere (quelli in neretto sono i 'verbs')   

Si possono definire policy molto granulari con la RBAC (motivo per cui ABAC è praticamente deprecato)  


### Aggregazione dei ruoli  

Possiamo definire dei ruoli che aggregano altri ruoli, combinandone le regole.   

A tal fine, definiamo un selector di quali ruoli aggregare, in questo caso le rules devono essere vuote.   

```yaml
apiVersion:rbac.authorization.k8s.io/v1
kind: CusterRole
metadata: 
  name: monitoring 

aggregationRule:
  clusterRoleSelectors:
  - matchLabels:
    rbac.example.com/aggregate-to-monitoring: "true"
rules: []
```  

il ruolo 'monitoring' è definito aggregando altri ruoli (clsterRoleSelector), e il selettore è matchLabels  

possiamo definire una serie di miniruoli, mettiamo a tutti la stessa etichetta dentro a metadata labels e poi li aggreghiamo in un parent.   

Solo i clusterRole possono essere aggregati!!   


**RoleBinding e ClusterRoleBinding**   

Questi ci permettono di associare le regole a un identità  

I role binding e clusterRoleBinding associano un ruolo o clusterRole con un soggetto, indicando quali policy applicare a tale soggetto  

guardare documentazione per vedere la struttura del YAML.  
in tale file `- kind:` mi dice a quale identità si riferisce   
la roleRef mi dice ruolo o clusterRole al quale i soggetti (definiti nel campo `subjects`)saranno intitolati   

Da un roleBinding possiamo applicare comunque un roleRef.kind: ClusterRole - possiamo farlo per questioni di namespace

### Soggetti in Kubernetes  

ServiceAccount è il primo soggetto. come se fosse un utente (non human).  
è una risorsa k8s (gestibile via API) indicante l'identità che un pod può assumere, si usa per definire come un Pod deve poter accedere all'APII di kubernetes

si gestisce come risorsa.  

### User

User sono gli utenti umani.  
Per K8s un user è una risorsa esterna al cluster, indicante un identità frutto di una procedura esterna di autenticazione.   

Un utente è identificato da una sola proprietà -> il nome   
l'api server fa configurato con un qualche metodo di auth.   

Gli utenti possono fare parte di Gruppi.  
Gruppi: un insieme di user, determinato in fase di autenticazione    

gruppi e utenti sono opachi per k8s.   

K8s definisce un gruppo speciale `system:masters`, che è il gruppo 'root' di un cluster k8s; hanno un ruolo "cluster-admin" che permette tutto.   

**Ruoli builtin**:   
- cluster-admin -> root 
- admin -> root, ma di un namespace 
- edit -> versine di admin che non permette di modificare ruoli e loro binding 
  - altrimenti admin potrebbe fare priviledge escalation e diventare cluster-admin 
- view -> permette di visionare tutte le risorse (tranne i secret) (versione ristretta di view)  
  - se riusciamo a guardare i sercret potrebbe guardare i token di auth e autenticarsi come un altro utente   



### Meta autenticazione   

**Facility di introspezione**: se l'utente può utilizzare l'API `SubjectAccessReview` il comando kubectl auth can-i permette di scoprire se una determinata azione è permessa



**Impersonamento**: si può prendere il ruolo di un altro utente/gruppo/serviceaccount usando il verbo `impersonate`, utilizzabile da kubectl tramite:
- `--as=<nome_utente>`
- `--as-group=<nome_gruppo>`   



### Metodi di autenticazione  

**X509 Client Authentication:**  
ogni cluster k8s ha un api-server che ha un certificato CA di cui si fida e quindi si può presentare per l'autenticazione un certificato X509 firmato dal API-Server CA e marcare che utente siamo e a che gruppi apparteniamo dentro al subject.  


**Service Account Tokens:**  
Quando si crea un service account k8s genera un token, che può essere usato dentro o fuori dal cluster. Questo token viene periodicamente ruotato (il token è un JWT).   
-> soluzione considerata "sporca"   

**OpenID Connect Tokens:**  
Si utlizza un token generato da un Identity Provider.  
Consiste nel modo pulito e consigliato.  

**Webhook**:  
Permette di far decidere l'identity a un servizio contattabile via API-Server 

**Reverse proxy**:   
Utilizza gli header di un reverse proxy per l'identificazione.  

**Static Tokens**:   
Token ed utenti definiti sull'API-Server in modo statici.   


### Autorizzazione:   

Elementi di una richiesta:  
- user -> utente 
- group -> gruppo 
- extra -> metadati forniti dall'autenticatore 
- verb -> cosa vogliamo fare nell'api-server; es. get, list, create, update, ...
- HTTP verb -> get, post, ...; la traduzione del verb in HTTP
- resource -> id della risorsa, es. id del pod, id della config map 
- subresource -> una sottorisorsa, es. i log del pod (es. vogliamo che uno possa fare la get per vedere i dettagli del pod ma che non possa fare la execute sul pod)   
- namespace 
- API group -> ""


Tipi di richieste:  

- Non-risorse: sono richieste agli endpoint a un path diverso da     
  - /api/v1 ;
  - /apis/<gruppo>/<versione>  

- Risorse: tutte le altre  

### Metodi di autorizzazione   

Kubernetes permette uno o più di questi metodi di autenticazione, se ne vengono usati molteplici, vince quello messo prima nella configurazione.  

- alwaysAllow 
- alwaysDeny
- ABAC
- RBAC
- Webhook   

Il default è RBAC

Gli autorizzatori vengono lanciati uno dopo l'altro e possono fornire un verdetto di: allow, deny, non definito (cioè l'autorizzatore non sa cosa fare)  

Se non sa cosa fare si passa all'autorizzatore dopo.  

Il default di K8s è RBAC, e se RBAC non sa cosa fare allora si passa a AlwaysDeny.   


### Admission Controllers   

All'interno della procedura di autenticazione possiamo inserire dei middleware che guardano e modificano le richieste fatte all'API-server, possono fare queste due operazioni:  

1. Validating: permette di controllare le richieste ed eventualmente rifiutarle 

2. Mutating: permette di modificare il contenuto (es. l'utente fa il deploy di un pod di un certo tipo, e possiamo controllare che un pod rispecchi delle policy e nel caso rifiutarlo)
