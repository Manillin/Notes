# Basics of Terraform   

Si crea una directory e un file main.tf con il seguente contenuto:

``hcl 
terraform {
  required_providers {
    docker = {
      source = "kreuzwerker/docker"
      version = "~> 3.0.1"
    }
  }
}

provider "docker" {}

resource "docker_image" "nginx" {
  name         = "nginx:latest"
  keep_locally = false
}

resource "docker_container" "nginx" {
  image = docker_image.nginx.image_id
  name  = "tutorial"
  ports {
    internal = 80
    external = 8000
  }
}
```



**Terraform Configuration**: L'insieme di file che descrivono l'infrastruttura in Terraform (può essere uno o più file modulari, terraform legge tutto insieme).   
Ogni terraform configuration deve essere nella sua working directory dedicata  


**Terraform Block**: Il `terraform{}` block contiene le impostazioni di terraform come i provider richiesti che tf userà per creare l'infrastruttura.  
Per ogni provider l'attributo `source` definisce un: hostname, namespace e provider type.  
Tf installa provider dal [terraform registry](https://registry.terraform.io/)    
Si può anche specificare una versione specifica, se omessa scaricherà di default la latest.  



**Providers**: Un provider è un *plugin* che tf usa per creare e gestire le risorse.  
Il blocco `provider` specifica il provider, in questo caso è `docker`.    
Possiamo usare più blocchi provider, e possiamo anche usare provider diversi per gestire risorse diverse dentro la stessa infrastruttura.


**Resources**: Usiamo i blocchi `resource` per definire componenti della nostra infrastruttura -> le risorse potrebbero essere fisiche o virtuali (come un container docker)   
I blocchi resource hanno due stringhe prima delle parentesi graffe:
- resource type: es. docker_image
- resource name: es. nginx  
Il prefisso del resource type viene mappato al provider (nell'esempio tf gestisce la risorsa 'docker_image' con il provider 'docker').   

Il resource type + resource name insieme formano un ID univoco per la risorsa, nell'esempio l'ID per l'immagine docker sarebbe: `docker_image.nginx`.     

I resource blocks contengono argomenti per configurare la risorsa (es. size macchina, immagini disco, ...).  
Leggere la [Providers Reference](https://developer.hashicorp.com/terraform/language/providers) per conoscere gli argomenti richiesti e quelli opzionali per ogni tipo di risorsa.  



### Inizializzare la Directory    

Quando creiamo una nuova configurazione, bisogna inizializzare la directory con `terraform init` .  
Questa operazione scarica e installa i provider definiti nella configurazione.  

Il provider scaricato viene installato nella working directory dentro una cartella nascosta chiamata `.terraform`
Questa fase di init crea anche:
- `terraform.lock.hcl`: un file che specifica l'esatta versione del provider usato, in questo modo scegliamo noi quando aggiornare i provider.  

**formatting e validation**:  

Per controllare che il formatting dei file sia giusto possiamo usare `terraform fmt` nella working dir, se non restituisce nulla vuol dire che il file è correttamente formattato.  

Per controllare che la configurazione sia valida e consistente possiamo usare `terraform validate` che restituirà un success o una failure.   


### Creare l'infrastruttura  

Per creare l'infrastruttura usiamo il comando 

```baash
terraform apply 
```

**terraform plan**: dopo l'apply tf stamperà nel terminale quello che andrà a fare per applicare l'infrastruttura definita, usando una notazione simile al version control di git.   
Questa fase non applica nessun cambiamento, chiederà una conferma prima di applicarlo.  
Una volta confermato il plan, l'infrastruttura verrà applicata.   


**Inspect State**:    

Quando applichiamo la configurazione, tf scrive dati nel file `terraform.tfstate` -> terraform state file.  
In questo file tf salva gli ID e le proprietà delle risorse che gestisce per poterle aggiornare ed eliminare in futuro.    

Il terraform state file è l'unico modo con cui tf gestisce le risorse e spesso contiene informazioni sensitive -> bisogna salvarlo in modo sicuro e dare accesso solo a persone fidate.   

`terraform show`: permette di ispezionare lo stato attuale 
`terraform state`: permette di controllare a mano lo stato, è un comando che ha sottocomandi come:
    - `terraform state list`: elenca le risorse nello stato

### Modificare l'infrastruttura    

Le infrastrutture non sono fisse e statiche ma dinamiche e in continua evoluzione, terraform permette di gestire le modifiche.   

Quando modifichiamo o cambiamo infrastruttura tf costruisce un *execution plan* che modifica solo ciò che è necessario per raggiungere lo stato desiderato.   

*Nota:* quando si usa tf in production è consigliato usare un sistema di version control per gestire i file di configurazione e salvare lo stato.   


Esempio di modifica:   
Prendiamo il main.tf e modifichiamo la risorsa `docker_container.nginx` cambiano l'attributo `ports.external` da 8000 a 8080.  

Questo update cambia il port number che il container docker sa per servire il server nginx, il provider docker sa che non può cambiare la porta di un container una volta creato, di conseguenza tf distruggera il vecchio container e ne creerà uno nuovo con la nuova porta aggiornata.   

Dopo aver cambiato la configurazione in main.tf lanciamo il comando `terraform apply` per vedere l'execution plan che tf propone per apportare le modifiche.   

Mostrerà una serie di attributi precedute da prefessi:
- `+`: crea l'attributo
- `-`: elimina l'attributo
- `~`: aggiorna l'attributo in place

L'execution plan mostra cosa forza tf a eliminare / modificare certi attributi (nell'esempio dice espressamente che il cambio di port forza la creazione di un nuovo container con la nuova port impostata)   

Una volta letto l'execution plane, se tutto è in ordine, scriviamo `yes` per apportare le modifiche.

### Destroy   

Il comando `terraform destroy` termina le risorse gestite dal progetto terraform.  
è l'inverso di `terraform apply` in quanto termina tutte le risorse specificate dal terraform state.  
NON distrugge risorse che sono in esecuzione altrove e che non sono gestite dall'attuale terraform project.

### Variabili  

Creare configurazioni con valori hardcoded non permette flessibilità, per questo motivo tf permette di creare configurazioni che includano variabili per renderle dinamiche e flessibili.   

Possiamo creare un nuovo file nella working directory chiamato `variables.tf` dentro al quale definiamo le nostre variabili.  

*Nota:* tf carica insieme tutti i file nella working directory che terminano in .tf, possiamo chiamare i nostri file come vogliamo.  

```hcl
# variables.tf 
variable "container_name" {
  description = "Value of the name for the Docker container"
  type        = string
  default     = "ExampleNginxContainer"
}
```

una volta fatto ciò, modifichiamo il main.tf come segue:

```hcl
resource "docker_container" "nginx" {
  image = docker_image.nginx.image_id
- name  = "tutorial"
+ name  = var.container_name
  ports {
    internal = 80
    external = 8080
  }
}
```

A questo punto quando facciamo l'apply possiamo passare la variabile come segue: 

```bash
terraform apply -var "container_name=NewName"
```
 
Tf supporta molti modi per usare e settare variabili, possiamo evitare di elencarle una ad una sul comando, per maggiori informazioni leggere [Customize Terrafom Configuration with Variables](https://developer.hashicorp.com/terraform/tutorials/configuration-language/variables)

### Outputs   

Possiamo creare valori di output pe organizzare i dati e renderli facilmente reperibili tramite query.   

Definiamo un file `outputs.tf` nella working dir (sempre per motivi di leggibilità e modularità) e scriviamo la seguente configurazione:  

```hcl
output "container_id" {
  description = "ID of the Docker container"
  value       = docker_container.nginx.id
}

output "image_id" {
  description = "ID of the Docker image"
  value       = docker_image.nginx.id
}
```

una volta fatto ciò, applichiamo la configurazione (o la modifichiamo se esisteva già) con `terraform apply ` e confermiamo con il `yes`.   

A questo punto, quando la configurazione è live possiamo usare il comando:  

```bash
terraform output
```

per fare la query sugli output, avremo in output i valori che avevamo impostato:

```bash
container_id = "ad139167bb76f9db5faef83e6ff9342179b089fdc7288abd3ac3c249c2abbe70"
image_id = "sha256:0cf1d6af5ca72e2ca196afdbdbe26d96f141bd3dc14d70210707cf89032ea217nginx:latest"
```
